2 · System architecture

How a program becomes a run. The golden rule: one deterministic event log is the only thing the views ever see.
INPUTS
Editor (CodeMirror 6)
Kid writes real JavaScript + repeat n {}. Friendly, line-pointed errors — errors never cost points.
Content (JSON)
Missions, parts, bot.json. Data-driven — new content never touches the engine.
↓
EXECUTION — the one and only code path
Web Worker Sandbox
Desugar repeat → transform to a generator → step it. Isolated from the page; per-tick step budget.
→
Sim Core
Deterministic, fixed ticks, seeded. Grid physics, commands, gates, scoring. Renderer-free.
→
EVENT LOG
The single output: move · turn · bump · honk · gateOpen · coin · score · clear. Same code → same log, always.
↓every consumer below is "dumb" — it replays the log, it runs no game logic
CONSUMERS (view layer)
Phaser 4 Arena
Vector bot + furniture; tweens the log. Supersampled, crisp.
Sound
Web Audio SFX per event — tread, turn, honk, clunk, win.
HUD
ARMOR + SCORE meters (Law 5: no number before it's felt).
Tank Radio
Friendly play-by-play the kid debugs by.
↓
Result + Save
Stars (complete · par-lines · bonus), coins, unlocks → localStorage. "Lines vs par" nudge, no timer.
Golden tests
Replay every author solution; assert the event log + final state — never pixels. That's why the view stays a dumb consumer.
Why this shape? Because the event log is deterministic and the sole execution path, we get ghosts & replays for free (a ghost is just a saved log replayed on a translucent bot), fair battles later, and a testable game where correctness never depends on rendering.
